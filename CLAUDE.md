## 절대 규칙
1. **계획 먼저**: 작업 전 계획이 모호하면 반드시 [호칭]에게 확인받아라
2. **자기 점검 필수**: 작업 완료 후 반드시 [검증 명령어 — 예: flutter analyze + flutter test] 통과 확인. 검증 없이 "끝났습니다" 하지 마라
3. **실수 기록 필수**: 작업 중 실수가 발생하면 `work-logs/work-log.md`에 즉시 기록. 작업 시작 전 확인하여 같은 실수를 반복하지 마라
4. **즉시 커밋 & 푸시**: 작업했으면 바로 커밋하고 **푸시까지** 완료해라. 푸시해야 배포된다
5. **수정 범위 제한**: 요청받은 것만 수정하라. 관련 없는 코드를 "개선"하지 마라. 새 파일은 꼭 필요할 때만 만들어라
6. **작업 완료 = 배포 완료**: 코드를 수정했으면 커밋 → 푸시 → 웹 배포까지 끝내야 "완료"다. 로컬에만 있으면 안 끝난 것이다

## 작업일지
- 작업 시작 전 `work-logs/work-log.md`를 읽고 이전 상태를 파악하라
- 작업 시작/완료 시 기록하라. 변경한 파일 경로를 명시하라
- 형식:
```markdown
## 2026-03-20
- **시작**: 무엇을 할 건지
- **완료**: 결과물 + 변경 파일 목록
- **이슈**: 발생한 문제 + 원인 + 해결 여부
```
